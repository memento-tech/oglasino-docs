# JPA Fetch Tuning

**Status:** built / pending verification
**Repos:** oglasino-backend only
**Branch:** dev

## Why this exists

The product read/search path is Elasticsearch, not the database (`DefaultProductsSearchService` reads `ProductDocument` via `ElasticsearchOperations`; the DB `Product` entity is touched only on single-row mutations and the batch reindex). So this feature is **not** a user-facing latency fix for the hot path — that path does not hit JPA.

The real justification is **connection-pool pressure**. The pool is deliberately capped at 18 connections against a DigitalOcean 1GB Postgres box (25-conn ceiling). Every avoidable N+1 multiplies the query count per request under concurrency, which is the failure mode behind the 2026-05-14 pool-exhaustion incident. This feature reduces query *count*, the correct lever for this box — not query latency, and explicitly not pool size.

Scope was set by a read-only audit (2026-06-03) plus four independent cold-session verification audits (one per finding cluster), archived to `sessions/`. Nothing here changes an HTTP contract or the Elasticsearch read path.

## Audited reality (ground truth from the verification audits)

- **No read-side fetch tuning is configured.** No `hibernate.default_batch_fetch_size` in any environment YAML; no `@BatchSize` anywhere. The only performance Hibernate properties present (`jdbc.batch_size`, `order_inserts`, `order_updates`) tune writes. (Briefs A, B, C — independently confirmed.)
- **`spring.jpa.open-in-view` is unset → Spring Boot default `true`.** OSIV is load-bearing: the read services and facades are mostly **not** `@Transactional`, and DTO mapping runs in the facade after the repository transaction closes. Several read paths rely on OSIV to initialize lazy associations at mapping/serialization time. See "OSIV dependency" below.
- **`Review.translations` is `FetchType.EAGER` but read by no consumer.** All converters fetch the single current-language row via `findByReviewAndLanguage`; the eager collection is loaded ~20×/page and discarded. (Brief B — verified no consumer navigates the collection.)
- **The `activeProducts` count subquery in the user-info projection is not fully index-covered.** `owner_id` is indexed; `product_state` is not. The sibling deletion-request subquery is already covered by `UNIQUE(user_id)` and needs nothing. (Brief D.)
- **The reindex hydrate query fetches three collections at once**, producing an `N × M × K` cartesian row set per product (translations × filterValues × imageKeys), × batch size 500. One bag among the three, so no `MultipleBagFetchException` today. (Brief C.)
- **Two request-path N+1s the original audit missed:** follower/following listing (~5 queries per user — baseSite, region, city, products collection, shortBio) and admin report listing (~4 per row). The follower listing is the heaviest N+1 in the codebase and is a user-facing endpoint. (Brief A.)
- **Per-row translation/shortBio queries are explicit repository calls, not lazy loads** — `getReviewTranslation` (review + owner-review listings) and `getUserShortBio` (follower listing). `default_batch_fetch_size` does **not** mitigate these; they need a batched-lookup code change.

## Batches

Each batch is an independently shippable engineering session. Order matters where noted.

### Batch 1 — global batch-fetch + Review.translations LAZY (config + annotation)

- Add `hibernate.default_batch_fetch_size` (50, mirroring `jdbc.batch_size`; 25 is equally defensible) to the JPA properties block in all three env YAMLs.
- Change `Review.translations` from `FetchType.EAGER` to `FetchType.LAZY`.
- These ship in the **same session**: the LAZY flip without batch-fetch would relocate the per-row load rather than collapse it. (For `Review.translations` specifically, no consumer reads the collection, so the flip is functionally free either way — but batch-fetch is what helps the *other* association N+1s, so they belong together.)
- No contract change, no ES impact. Existing `./mvnw test` suite must stay green.

### Batch 2 — activeProducts partial index (in-place V1 edit)

- Add `CREATE INDEX idx_product_owner_active ON public.product (owner_id) WHERE product_state = 'ACTIVE';` to `V1__init_schema.sql`, alongside the existing partial-index precedents (`idx_product_deactivated_by_system`, `idx_udr_pending_due`, `idx_users_deletion_status`).
- **Partial form only.** Both current call sites bind `ProductState.ACTIVE`. If a future caller parameterizes the state, the planner falls back to `idx_product_owner` + residual filter — acceptable. Do **not** add the composite `(owner_id, product_state)` form; do **not** add the deletion-request index (already covered by `UNIQUE(user_id)`).
- Predicate is column-to-string-literal → IMMUTABLE-safe (Part 12).
- **Accompanying edit, same session:** correct the `V1__init_schema.sql` header. It currently says "Do NOT edit this file after it has been pushed … add a new V{n} file," which contradicts `conventions.md`'s pre-launch "edit V1 in place" rule. Per Igor's decision (2026-06-03), pre-launch schema work edits V1 in place; update the header to say so, so the file stops contradicting the rulebook.
- Edits V1 because the schema is rebuilt from a single V1 on every dev/stage reset pre-launch (`ddl-auto: validate`).
- Low urgency: the subquery runs only on a `redisUserInfo` cache miss, bounded by per-owner product count.

### Batch 3 — reindex collection split + dead-method deletion

- The reindex mapper (`DocumentProductConverter`) reads only `translations` and `imageKeys` off the `Product` entity. Filter references come from a **separate per-product query** — `ProductFilterValueRepository.findByProductId`, which carries its own entity graph (filter / filterRange / filterOptions) — and **never** from `product.filterValues`. The `filterValues` (and `fv.filter`) join fetches in the hydrate query were therefore dead weight, not a needed collection.
- Batch 3 **removed those `filterValues` join fetches entirely** (rather than splitting them into a second query) and **split the two remaining collections**: `findForIndexingByIds` fetches one `Set` (`translations`), and a companion query over the same page of IDs fetches the other `Set` (`imageKeys`), merging into the managed entities within the same persistence context. The mapper at `toIndexQuery` must still see fully-populated collections — verify reindexed documents stay complete.
- Delete the dead `findAllForIndexing()` (zero callers in `src/main` or tests; a non-paginated full-graph cartesian fetch that is an OOM footgun if revived).
- **Outcome:** the hydrate query now fetches **zero bags** (two `Set`s, split across two queries, `filterValues` not fetched at all), so `MultipleBagFetchException` is impossible by the *absence of any bag* — not merely by the single-bag rule. The `Set`-not-`List` typing of `translations` and `imageKeys` remains the invariant that keeps it so.
- **Requires a reindex smoke** (admin `/es/reindex` against a product with multi-row collections) before close.
- Batch job only, not request path — correctness-safe but the largest heap win during a full reindex.

### Batch 4 — batched translation/shortBio lookup across the review + follower listings

Replace the per-row translation/shortBio queries with a page-bounded batched lookup, across the surfaces that actually carry the N+1:

- **Three review-side surfaces** sharing one batched `ReviewTranslation` lookup: the public review listing (`ReviewConverter`), the owner-review listing (`OwnerReviewConverter`), and the **admin review listing** (`AdminReviewConverter` / `DefaultAdminReviewFacade`) — the last was *added* to scope once it proved to carry the identical per-row `getReviewTranslation` N+1.
- **The follower/following listing** (`EntityUserInfoConverter`), using one batched `UserTranslation` (shortBio) lookup. (The baseSite/region/city/products lazy loads on this path are batch-fetched by Batch 1; the shortBio query is not.)
- **Admin report listing is NOT in scope** — *dropped* from Batch 4. It has no per-row translation/shortBio query; its reporter/reportedUser + baseSite lazy-load N+1 was already collapsed by Batch 1's `default_batch_fetch_size`.

Implementation shape:

- One batched query method per repository (`findByReviewIds` / `findByUserIdsAndLanguageId`-style), page-bounded, fed to the ModelMapper converters via a thread-local mapping context mirroring the existing `UserOverviewMappingContext` precedent. **One shared *pattern*, not one shared *method*** (Part 4a) — the two paths hit different tables, return types, and missing-row semantics, so a single forced helper would be worse than the shared pattern.
- The single-id `getReviewTranslation` was deleted (no callers after migration); the single-id `getUserShortBio` was retained (it has non-listing callers).
- The heaviest request-path N+1 in the codebase (follower listing) is here. This is a real code change, the largest session, and goes last.

**Closeout behavior change — graceful missing-translation fallback.** A review missing a translation row for the reader's current language previously made the listing converter throw, which 500'd the entire review page. This is a reachable production path: review creation generates the non-author-language rows inside an OpenAI call whose exceptions are swallowed, so an OpenAI outage at creation time persists a review with only the author's-language row. The closeout session changed the batched lookup to fetch **all** language rows for the page's reviews in one query and resolve in memory — current language, else the original (human-written, non-auto-translated) row, else any row, else empty fields. Review listings now render gracefully instead of 500ing on a missing row. This is a deliberate, recorded behavior change. The follower/shortBio path was unchanged (it already returned an empty string on a missing row).

### Parked — Product.imageKeys EAGER → LAZY

Not in scope. The flip saves one cheap indexed fetch on cold mutation paths, but the careless version silently breaks the image-deletion sweep (`deleteProduct` reads `imageKeys` on a virtual thread off a closed transaction; the `LazyInitializationException` is swallowed and images stop deleting with no error), and it needs an accompanying entity-graph on the owner edit-form GET or it adds a query there. Marginal upside, silent-failure downside, no free ride. Revisit post-launch if image-table growth makes the eager fetch measurable.

## OSIV dependency (cross-feature hazard)

`spring.jpa.open-in-view` defaults to `true` and is load-bearing on several read paths — `Review.imageKeys`, the follower listing, and the admin-report listing all initialize lazy associations during DTO mapping *after* the service transaction has closed, surviving only because OSIV keeps the session bound to the request thread.

This dependency exists **today, independent of this feature.** Batch 1's `Review.translations` flip does **not** add to it (no consumer traverses that collection). But the active `backend-security-hardening` track must not disable OSIV without first landing fetch-joins / entity graphs / batch-fetch on these paths, or those endpoints throw `LazyInitializationException` at serialization. Recorded as a `state.md` Risk Watch entry at feature close.

## Trust boundaries (Part 11)

No impact. These are read-only listing/projection paths; the values used (`approved = true`, owner id from `currentUserService.getCurrentUserIdStrict()`, the bound `ProductState.ACTIVE`) are server-derived, not client-supplied. No recommendation moves a decision off the server.
