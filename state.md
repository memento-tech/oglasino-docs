## Feature pipeline

### Feature statuses

- `planned` — spec exists, no engineering started
- `in-progress-backend` — backend work active or required next
- `backend-stable` — backend done, contract frozen, web can adopt
- `in-progress-web` — web work active
- `web-stable` — web done, mobile can adopt
- `in-progress-mobile` — mobile work active
- `mobile-stable` — all three platforms shipped to feature branch
- `shipped` — merged to main, in production (or staging if pre-launch)
- `parked` — deliberately deferred

- **Firebase auth emails still ship from `firebaseapp.com` (accepted-and-known until post-launch).** Registration verification, password reset, and email-change confirmation emails currently arrive From `noreply@<project>.firebaseapp.com` because the 2026-05-22 Brevo SMTP integration only wired backend-originated transactional email. Igor's decision per the 2026-05-22 decisions.md entry is Option C (full unification on Brevo via Firebase Admin SDK link-generation methods) as the target end-state, but the unification work is deferred to a post-launch Mastermind chat. Handoff brief at [`.agent/handoffs/email-followups.md`](.agent/handoffs/email-followups.md) carries the Option C plan. Close this row when Option C ships.
- **Non-essential cookies set without consent gating — CLOSED (2026-05-21, Consent Mode v2 Brief 6).** Card-size cookies (`globalCookie.portalCardSize`, `globalCookie.dashboardCardSize`) now gate their write on `og_consent.preference === 'granted'` via `isPreferenceConsentGranted()` in `src/lib/consent/gating.ts`. The `userPreferenceService` tracking-cookie surface was surfaced as an adjacent finding and queued in the same handoff brief for removal.
- **Shared-browser cross-user consent contamination (accepted in v1).** `og_consent` is browser-local with no per-user scoping; a logged-out user on a shared browser inherits the prior user's consent state.
