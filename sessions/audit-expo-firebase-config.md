## 6. Environment separation analysis

### Does `oglasino-expo` have separate stage and prod configurations today?

**Partial тАФ structurally present but substantively broken.**

The separation mechanism exists:

- `app.config.ts` branches on `APP_ENV` to select different bundle IDs, native config file paths, and API URLs.
- `eas.json` defines `development` and `production` build profiles that set `APP_ENV` accordingly.
- `.env.development` and `.env.production` are separate files with the right variable names.

But the implementation is broken in three ways:

1. **`.env.production` has all Firebase values empty.** A production EAS build would initialize Firebase with `undefined` for every config field тЖТ `initializeApp` would throw or produce an inoperable app.

2. **Production native config files don't exist.** `app.config.ts` references `google-services.prod.json` and `GoogleService-Info.prod.plist` but neither file is in the repo. A production EAS build would fail during the Expo prebuild step when trying to copy these files.

3. **Development points to a third project (`oglasino-dev`), not to stage.** Even the working development path doesn't target the right Firebase project.

### What patterns do `oglasino-web` and `oglasino-backend` use?

Per `state.md` and the GA4 feature:

- Web uses `NEXT_PUBLIC_GA4_MEASUREMENT_ID` and other env vars set per Vercel environment.
- Backend uses Spring profiles (`application-dev.yaml`, `application-stage.yaml`, `application-prod.yaml`) with per-profile Firebase project config.
- Both target `oglasino-stage-49abb` for stage and `oglasino-prod-7e5db` for prod.

### What would the split need to look like?

(Inventory only тАФ no implementation proposed per the brief.)

Missing pieces:

1. `.env.development` values need to be updated to `oglasino-stage-49abb` credentials.
2. `.env.production` values need to be populated with `oglasino-prod-7e5db` credentials.
3. `google-services.dev.json` needs to be regenerated from the `oglasino-stage-49abb` Firebase console for `com.oglasino.dev` Android app.
4. `GoogleService-Info.dev.plist` needs to be regenerated from `oglasino-stage-49abb` for `com.oglasino.dev` iOS app.
5. `google-services.prod.json` needs to be created from `oglasino-prod-7e5db` for `com.oglasino` Android app.
6. `GoogleService-Info.prod.plist` needs to be created from `oglasino-prod-7e5db` for `com.oglasino` iOS app.
7. An `.env.example` documenting the schema would be valuable (web has `.env.local.example`).
8. `EXPO_PUBLIC_FIREBASE_MEASUREMENT_ID` needs to be defined per environment if analytics is intended (stage: `G-P0LEVEJ0V9`, prod: `G-GNKB4WBNC0`).

## 7. Cross-check against ╬ж1's auth wiring

### Which project ID flows into Firebase Auth initialization?

The auth path:

1. `src/lib/client/firebaseClient.ts:11-17` reads `process.env.EXPO_PUBLIC_FIREBASE_*`
2. In development: resolves from `.env.development` тЖТ `projectId: 'oglasino-dev'`, `authDomain: 'oglasino-dev.firebaseapp.com'`
3. `initializeApp(firebaseConfig)` at line 29 creates a Firebase app targeting `oglasino-dev`
4. `initializeAuth(app, ...)` at line 31 creates auth against `oglasino-dev`
5. `authStore.ts:197` `onIdTokenChanged(auth, ...)` тАФ ╬ж1's auth listener subscribes on the `oglasino-dev` auth instance
6. `authService.ts:218` `syncUserToBackend(firebaseUser)` sends the Firebase ID token to the backend
7. The backend's `FirebaseAuthFilter` verifies the token against its configured Firebase project

### Verdict: **HIGH SEVERITY**

Mobile authenticates against `oglasino-dev`. Backend and web authenticate against `oglasino-stage-49abb` (stage) or `oglasino-prod-7e5db` (prod).

**Firebase ID tokens are project-scoped.** A token issued by `oglasino-dev` will be **rejected** by a backend configured to verify tokens from `oglasino-stage-49abb`. The backend's `FirebaseAuthFilter` calls `FirebaseAuth.getInstance().verifyIdToken(token)` тАФ if the Firebase Admin SDK is initialized with `oglasino-stage-49abb` credentials, it will reject tokens from `oglasino-dev` as invalid.

**Implications:**

- **╬ж1's device smoke is invalid** unless Igor's local backend happens to also be pointed at `oglasino-dev`. If the local backend uses `oglasino-stage-49abb` (which the stage configuration suggests), then `syncUserToBackend` will receive 401s on every call because the token is from the wrong project.
- **Firestore data isolation:** Chats, notifications, and user data written by mobile go to `oglasino-dev`'s Firestore, not to the Firestore that web and backend share. Mobile users would see no chats, no notifications, and none of their data from the web.
- **FCM token isolation:** Push notification tokens registered on mobile would be in `oglasino-dev`'s FCM project. The backend sending push via `oglasino-stage-49abb`'s Admin SDK would fail to deliver to these tokens.

**Caveat:** if the local development backend (running on the IP in `.env.development`) is also configured to use `oglasino-dev`, the auth flow works locally. But this means Igor has been developing against a completely separate Firebase project from the one web uses тАФ separate user pool, separate Firestore, separate FCM. This may be intentional for isolated mobile development but is incompatible with the cross-platform data-sharing requirement.

## Adjacent observations (Part 4b)

### A1 тАФ `firebaseAnalytics.ts` uses web pattern `typeof window !== 'undefined'`

- **File:** `src/lib/client/firebaseAnalytics.ts:8`
- **Severity:** low
- **Detail:** The `typeof window !== 'undefined'` guard is a Next.js/SSR pattern. In React Native with Hermes, `window` may or may not be defined. The check may not do what's intended (gate analytics to client-side only). The correct RN guard would use `Platform.OS` or simply omit the check since RN has no server-side rendering.
- Out of scope for this audit.

### A2 тАФ Dead `testConnection` function with `console.info`/`console.error`

- **File:** `src/lib/client/firebaseClient.ts:38-52`
- **Severity:** low
- **Detail:** Dead code (`// testConnection()` at line 52). Contains `console.info` and `console.error` which would violate Part 4 cleanliness if uncommented. ╬й cleanup target.
- Out of scope for this audit.

### A3 тАФ `.env.production` API URL uses HTTP, not HTTPS

- **File:** `.env.production:3`
- **Severity:** medium
- **Detail:** `EXPO_PUBLIC_API_URL=http://oglasino.com/api` uses plain HTTP. If this value is used in production, API requests would be unencrypted. iOS App Transport Security blocks plain HTTP by default unless explicitly exempted. The Cloudflare router would redirect HTTP тЖТ HTTPS, but the initial request would still be unencrypted in transit. Should be `https://oglasino.com/api`.
- Out of scope for this audit.

### A4 тАФ No `@react-native-firebase` packages

- **File:** `package.json:57`
- **Severity:** informational
- **Detail:** The app uses `firebase: ^12.10.0` (JS SDK), not `@react-native-firebase/*` (native SDKs). The JS SDK works in RN via the Expo managed workflow, but lacks native-only features like background messaging, crashlytics, and native analytics. This is an architectural decision, not a bug.
