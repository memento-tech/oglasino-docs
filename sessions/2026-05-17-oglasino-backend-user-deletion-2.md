## Implemented

- **Enum promotion preamble.** Promoted `UserDeletionRequest.status` from `String` to a new `DeletionRequestStatus` enum (`PENDING`, `CANCELLED`, `COMPLETED`, uppercase per Java enum convention). Updated the V1 `user_deletion_requests` CHECK constraint and DEFAULT to uppercase values, and the `idx_udr_pending_due` partial-index predicate to `WHERE status = 'PENDING'` (still IMMUTABLE-safe — string literal comparison). Updated `UserDeletionRequestRepository.findPendingDueByDate` and `deleteCompletedOlderThan` to JPQL form referencing the FQ enum (`com.memento.tech.oglasino.entity.DeletionRequestStatus.PENDING` / `.COMPLETED`). `cancelled_reason` stays a `String` per brief — its values `'user_login'` and `'admin_override'` are written rarely and only by the service layer; no enum promotion needed yet.
- **Phase 5.1 — `UserAuditService` + `DefaultUserAuditService`.** Five methods: `hashEmail` and `hashUserId` (both SHA-256 → lowercase hex64), `isEmailBanned`, `recordBannedUser` (idempotent on email-hash collision: returns existing row's id rather than inserting), `removeBannedUserRecord`. `recordBannedUser` reads `user.deletion.audit.retention.banned.months` from `ConfigurationService` and computes `retention_until` via `Instant → OffsetDateTime(UTC).plusMonths(...) → Instant` (Instant doesn't accept variable-length units directly, so the project-to-OffsetDateTime is required for correct calendar-month arithmetic).
- **Phase 5.2 — `FirestoreUserService` + `DefaultFirestoreUserService`.** Single method `deleteUserNotifications(firebaseUid)` — pages through `notifications/{uid}/userNotifications` in 500-doc Firestore batches (the SDK's batch-write limit), commits each batch synchronously, then deletes the parent `notifications/{uid}` document. Wrapped in a top-level try/catch; on any Firestore error logs at ERROR and returns (best-effort per spec §17.8 / §8.5).
- **Phase 5.3 — `FirebaseUserService` extension.** Three new methods on the existing admin-side interface: `deleteFirebaseUser`, `userExistsInFirebase`, `revokeRefreshTokens`. All three idempotent on `AuthErrorCode.USER_NOT_FOUND` (log at INFO and return); any other `FirebaseAuthException` is wrapped in `RuntimeException` per the existing service style.
- **Phase 5.4 — `ProductService` extension + refactor.** Took the recommended refactor: extracted the state-change body of `changeProductState` into a private `applyStateChange(product, state)` helper, made the public `changeProductState` perform the ownership check then delegate, and added the new public `changeProductStateAsSystem` which delegates without the ownership check. Both paths share the exact same DB save + `evictUserInfoCache` + `ProductStateUpdateEvent` publication, so the asynchronous ES reindex behaviour stays identical between the two callers. Also added `findProductIdsByOwnerAndState(ownerId, state)` to `ProductService` + a corresponding `findIdsByOwnerIdAndProductState` JPQL query to `ProductRepository`. The refactor is a deliberate cleanup beyond the strict ask; flagged in For Mastermind.
- **Phase 5.5 — `UserDeletionService` + `DefaultUserDeletionService`.** The largest piece. Seven public methods per spec §8.1: `requestDeletion`, `cancelDeletionOnLogin`, `executeScheduledDeletion`, `markForImmediateDeletion`, `forceDeleteByAdmin`, `lockFromDeletion`, `unlockFromDeletion`. The shared 16-step hard-delete routine is in private `runHardDelete(user, triggeredBy)`, called by `executeScheduledDeletion`, `markForImmediateDeletion`, and `forceDeleteByAdmin`. DB-side work (steps 1-12) runs inside a `TransactionTemplate.executeWithoutResult` block; external + audit-completion work (steps 13-16) runs after the transactional block returns, each wrapped in a `bestEffort(label, op)` helper that catches and logs without propagation. The `HardDeleteContext` local data class carries the captured pre-delete state (firebase UID, profile image key, was-banned flag, audit-log id, collected product image keys) across the transactional boundary. Per spec §3 reports handling: `linkReportsToBanIfBanned(user)` is the shared helper invoked from step 2 — if the user was disabled at deletion time AND a `banned_user_audit` row exists, reports are linked-and-anonymized; otherwise reports are deleted outright (defensive log if disabled-but-no-audit-row, falls through to delete). `markForImmediateDeletion` runs the §3.12 abuse check (open reports against ≥ 1 OR authored > threshold) and inserts a `banned_user_audit` row via `recordBannedUser` *before* invoking the hard-delete routine when the user wasn't already disabled. `forceDeleteByAdmin` looks up the user, checks for an active lock, then invokes the routine with `triggered_by='admin_action'`.
- **Phase 5.5 — three custom exceptions.** `UserDeletionException` is a new abstract base in the `exception/` package. Three concrete subclasses: `UserLockedFromDeletionException` (403 + `USER_LOCKED_FROM_DELETION` + `user.locked.from.deletion`), `ReauthRequiredException` (403 + `REAUTH_REQUIRED` + `errors.reauth.required`), `UserNotPendingDeletionException` (400 + `USER_NOT_PENDING_DELETION` + `user.not.pending.deletion`). Each subclass carries its own HTTP status, code, and translation key as abstract-method overrides — no enum, kept the surface minimal. Registered with `GlobalExceptionHandler` via a single new `@ExceptionHandler(UserDeletionException.class)` that unpacks the carried status + code + translation key into the existing unified `ProductErrorResponse` envelope (the wire shape is generic across `String` codes; no separate response class needed). Deliberately kept the codes out of `ProductErrorCode` per the open `issues.md` HIGH entry "ProductErrorCode is a dumping ground for cross-cutting error codes" — adding to that enum would extend the smell.
- **Phase 5.6 — `DefaultUsersFacade` extension.** Modified `disableUser(Long userId)` → `disableUser(Long userId, String banReason)` per spec §8.6. Backward-compat: the existing GET endpoint at `/api/secure/admin/users/disable/{targetUserId}` passes `null`; the facade defaults null/blank reasons to a placeholder `"admin disabled (no reason provided)"`. Order per spec §8.6: `saveUser` first (fires both Redis evictions per the 2026-05-14 connection-pool constraint), then `userAuditService.recordBannedUser` (idempotent), then `firebaseUserService.disableUser` + `revokeRefreshTokens`. `enableUser` mirrors: clear `disabled`/`banReason` via `saveUser`, then `removeBannedUserRecord` (FK cascade drops linked reports), then `firebaseUserService.enableUser`.

## For Mastermind

- **TransactionTemplate vs @Lazy self-injection — design decision in `DefaultUserDeletionService`.** Spec §8.1 and the brief both require the hard-delete sequence to commit its DB work before the R2/Firebase external calls run, so external failures cannot roll back the user delete. The two viable Spring patterns are `@TransactionalEventListener(AFTER_COMMIT)` and "split into a `@Transactional` inner method + a non-transactional outer". The brief allowed either. I rejected `@TransactionalEventListener` because it splits the work across two methods at distance (the event class plus a listener method), and the post-commit work has no business meaning outside this exact flow — making it an event would create a one-listener-only event class. Of the "split into inner + outer" patterns, the cleanest was `TransactionTemplate.executeWithoutResult(...)` — it makes the transactional boundary explicit at the call site (`writeTx.executeWithoutResult(status -> { ... })`) and avoids the `@Lazy self` indirection. The 2026-05-17 `DefaultBaseCurrencyService` precedent uses `@Lazy self.getBaseCurrency()` but the use case there is different (caching-proxy-aware self-call), not transactional-boundary-aware. Recommended for Mastermind: confirm `TransactionTemplate` is acceptable as a documented alternative pattern in this codebase. If `@Lazy self` is the only blessed style, the service refactors cleanly — extract the inner-transactional work to a public method, expose it on the interface, swap `writeTx.executeWithoutResult` for `self.doHardDeleteTransactional`. Either pattern produces identical runtime behaviour.
- **Separate exception hierarchy for user-deletion error codes.** Per the open `issues.md` HIGH entry, `ProductErrorCode` should not gain new cross-cutting codes. I created a new `UserDeletionException` abstract base with three concrete subclasses, each carrying its own HTTP status, code, and translation key as abstract-method overrides (no `UserDeletionErrorCode` enum — for three values, an enum-and-mapper layer would be overkill per Part 4a). `GlobalExceptionHandler` got a single `@ExceptionHandler(UserDeletionException.class)` that unpacks into the existing unified `ProductErrorResponse` envelope. The wire shape is unchanged; only the in-process exception hierarchy is separate. Recommended for Mastermind: confirm this pattern is the right precedent for future cross-cutting exception groups, OR confirm the "real fix" (refactor cross-cutting codes out of `ProductErrorCode` into a proper `SystemErrorCode` enum) gets its own brief.
- **Adjacent observation — `DefaultProductService.applyStateChange` refactor is a deliberate cleanup.** The brief flagged this as "Recommended (not required) refactor: have the existing `changeProductState` delegate to `changeProductStateAsSystem`". I took it: extracted the shared body into a private `applyStateChange(product, state)` helper, made `changeProductState` perform the ownership check then call the helper, and `changeProductStateAsSystem` call the helper directly. Both share identical save + evict + event-publish. Pre-existing test coverage on `DefaultProductServiceTest` (15 tests) continues to pass, which verifies the ownership-check path still works as before. Severity: low. Worth flagging because it touches existing code beyond the strict "add two methods" ask.
- **Adjacent observation — `Report.reporter` / `Report.reportedUser` linking via UPDATE works for the spec, but the entity model still treats them as required `@ManyToOne` without `optional=false`.** Spec §3 reports treatment relies on setting `reporter = NULL` and `reportedUser = NULL` via the JPQL UPDATE in `ReportRepository.linkAndAnonymizeReports*`. This works at the DB level (schema permits null) and at the JPQL level (no `optional=false` constraint). However, the `Report` entity does not declare these fields as nullable in any documentation; future code reading the entity could mistakenly assume they're always non-null and `@NonNull`-annotate them. Severity: low. Worth adding `// nullable for anonymized reports linked to banned_user_audit per features/user-deletion.md §6.3` Javadocs to both fields in a cleanup pass.
- **Adjacent observation — `BaseEntity.updatedAt` is `LocalDateTime`, but `UserDeletionRequestRepository.deleteCompletedOlderThan` accepts `LocalDateTime cutoff`.** This is correct (BaseEntity-inherited field) but inconsistent with the rest of the user-deletion entities which use `Instant` for their tz-aware timestamps. Phase 7's scheduled job will need to compute `cutoff` as `LocalDateTime.now().minusDays(retentionDays)` not `Instant.now().minus(...)`. Flagged so it's not missed at the call site. Already flagged in session-1's summary but worth re-flagging here because the call site is now closer (Phase 7).
- **Adjacent observation — Section 8.1 of the spec lists "step 15: Update user_deletion_requests: status='completed'", but `user_deletion_requests.user_id` has `ON DELETE CASCADE` on the `users` FK.** When step 12 deletes the user row, the cascade immediately deletes the deletion-request row. So step 15 is effectively a no-op — the row is gone before any post-step-12 code can update it. I implemented step 15 *before* step 12 (status flip happens, then cascade kills the row, then the in-memory snapshot reflects the COMPLETED status for any code that read it inside the transaction). The functional outcome is the same. The audit-purge job's `deleteCompletedOlderThan` query (Phase 7) is therefore effectively dead code — no row ever survives the cascade to be matched by `status = COMPLETED`. Mastermind may want to revisit: either (a) remove the cascade and rely on the audit-purge job to clean up, or (b) accept that the request table is ephemeral and have the audit-purge job target only the *audit log* (which it already does separately). I implemented (b) — request table is ephemeral, purge job targets audit log + ban audit. This matches the brief's Phase 7 job which has three separate deletes (audit log, ban audit, completed requests) but the third one will never find rows to delete. Spec inconsistency flag: spec §6.4 (cascade) + §8.1 (step 15) + §9.3 (purge completed requests) are mutually inconsistent. Recommended spec correction: drop step 15's "status='completed'" requirement (the cascade handles cleanup) and drop the `deleteCompletedOlderThan` call from the audit-purge job (it's dead code given the cascade).
