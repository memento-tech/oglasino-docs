### 2. The cookie consent banner

- **`CookieBanner` component exists.** `src/components/client/CookieBanner.tsx:19-93`. Status: **present**. Severity: **blocking**. Why: it covers two categories — `necessary` (always on, disabled toggle) and `preference` (user-toggleable). It does **not** cover an analytics/marketing category. Consent Mode v2 needs at minimum an `analytics_storage` decision; today there is no field for it.
- **Banner mount points.** Rendered at `app/[locale]/(portal)/layout.tsx:1,20` (portal segment) and `app/[locale]/owner/layout.tsx:2,44` (owner dashboard). Status: **partial**. Severity: **friction**. Why: the admin segment (`app/[locale]/admin/`) does not mount `CookieBanner`, so a user landing on an admin URL without prior consent does not see it. For analytics this is low-risk (admin is auth-gated and not a public surface), but Consent Mode v2 modeling requires the consent state to exist for every page-view event.
- **Consent persistence: cookie.** `src/lib/service/oglasinoCookies.ts:7-44` writes to a `globalCookie` document.cookie blob keyed by `cookieConsent` (`{necessary, preference}`). 365-day TTL. Status: **present**. Severity: **friction**. Why: there is no Cookiebot / OneTrust / custom CMP integration — it is a homegrown writer with no consent log, no audit trail, and a single bucket. GA4 Consent Mode reads `analytics_storage` / `ad_storage` / etc.; the existing schema does not include these, so the feature must extend `GlobalCookie.cookieConsent` or layer a new key.
- **Authenticated-user consent mirror.** `src/components/client/CookieBanner.tsx:48-52` calls `updateUserAuth({ allowPreferenceCookies })` on accept, mirroring the cookie decision into the backend user record. `app/[locale]/owner/user/page.tsx:52,113,185,277-283` surfaces this in the owner settings page as a single "preference cookies" toggle. Status: **partial**. Severity: **friction**. Why: only one consent dimension is mirrored; analytics consent is not modeled in the backend user record at all. The "Manage cookie preferences" UI for analytics consent will need backend coordination if Mastermind wants signed-in users' decisions to persist across devices.
- **No footer "Manage cookie preferences" link.** Grep of `src/components/server/layout/Footer.tsx` for `cookie` / `Cookie` returns nothing. Status: **absent**. Severity: **friction**. Why: the brief framing says `state.md` references this as a pre-launch action item; in code, only the owner settings page can change the preference after the fact, and an unauthenticated user has no surface to revisit the decision.
- **Translation coverage.** `tCookies('banner.header' | 'banner.description' | 'banner.required.label' | 'banner.preference.cookies' | 'banner.accept.label')` flow through `TranslationNamespaceEnum.COOKIES`. The audit did not enumerate the four-locale SQL seed coverage (out of scope per brief framing — the namespace exists and is wired). Status: **present**. Severity: **nice-to-have**. Why: new analytics-consent keys will need backend SQL append per conventions Part 6 Rule 3.

### 3. Identity — who is the user

- **Auth store and resolution.** `src/lib/store/useAuthStore.ts:129-332` is the single source of the authenticated user. `useAuthResolved()` at `src/lib/hooks/useAuthResolved.ts:13-27` is the existing three-state gate (Firebase ready + store-user synced). Status: **present**. Severity: **n/a**.
- **Stable non-PII identifiers available.** `AuthUserDTO` carries both `id` (backend integer userId) and `firebaseUid` (used as `senderId`/`receiverId` in chat — see `src/messages/store/useChatStore.ts:391`). Either is safe to send as GA4 `user_id`. Email is PII and must not be sent raw. Status: **present**. Severity: **n/a**.
- **Auth listener mount.** `src/components/client/initializers/AuthInit.tsx:9-15` mounts `initFirebaseAuth` at `app/[locale]/layout.tsx:39`. The actual onIdTokenChanged listener lives in `src/components/client/initializers/UseTokenRefresh.tsx` (mounted via `AppInit.tsx:28`). Status: **present**. Severity: **friction**. Why: `useAuthStore.initAuthListener()` (line 288-301) sets the store user only after `syncUserToBackend()` succeeds — there is a window (Firebase auth restored, backend sync in flight) where events fired by other components will lack `user_id`. Why: GA4 generates its own `client_id` from `_ga` cookie; no app-level work needed unless cross-device unauth identity is desired (it isn't, per the locked decisions).

### 6. PII surface analysis

- **`sign_up` payload.** `registerUserFirebase(email, password)` (`src/lib/service/reactCalls/authService.ts:179-184`) sends raw email and password to Firebase. After success, `syncUserToBackend()` POSTs `{ allowPreferenceCookies, … }` to the backend. The frontend has the raw email in scope. **PII flag:** `email` is raw PII — must not be sent to GA4. Safe alternatives: `firebaseUid` (post-create), backend `id` (post-sync), or hashed email. Status: **present**. Severity: **friction**. Why: the temptation to include "method=email" is fine; including the email itself would violate GA4's PII rule.
- **`contact_seller_clicked` payload.** The handler receives `withUser: UserInfoDTO` (`ProductFunctions.tsx:26-27`) which exposes `{ id, firebaseUid, displayName, profileImageKey, allowPhoneCalling }`. **PII flag:** `displayName` is user-published, not PII in the GA4 sense — but it is identifying. Safe alternative: `firebaseUid` or `id`. Email and phone number are not in `UserInfoDTO`'s shape, so they are not at risk of accidental inclusion. Status: **present**. Severity: **nice-to-have**.
- **`message_sent` payload.** `useChatStore.sendMessage(chatId, content)` (`useChatStore.ts:372`). The wire payload (`src/messages/types/chat/MessageRequest.ts`) carries `{ content, senderId, receiverId, productId?, seen, createdAt }`. `senderId` and `receiverId` are Firebase UIDs (safe). `content.blocks[].text` is user-typed and may contain a phone number, address, or other PII — **do not send to GA4**. Safe parameters: `recipient_id` (firebaseUid), `product_id`, `is_first_message_in_chat` (derivable from `isNewChat` at line 391). Status: **present**. Severity: **friction**.
- **`product_create_completed` payload.** `createNewProduct(productData, …)` (`UploadedProductDialog.tsx:52`) sends `{ name, description, price, currency, filters, imageKeys, topCategory, subCategory, finalCategory, regionAndCity, … }`. **PII flag:** `description` is user-published; per the brief, that's not PII in the GA4 sense even if it contains a phone number the seller chose to publish. **Safe parameters:** category id (topCategory/subCategory/finalCategory), price bucket, currency, image count — never raw name/description. Status: **present**. Severity: **friction**.

### 7. Performance and SSR considerations

- **GA4 script home.** `app/layout.tsx:32-51` is the root server layout; `<head>` is generated by `generateMetadata`. The `<body>` mounts `FirebaseWorkerInit`, `ConfigProvider`, `ThemeProvider`. A `next/script` for `gtag.js` would natively go in the root layout (server-rendered) or as a child client component. Status: **absent** (no current script). Severity: **friction**. Why: no precedent — the feature spec sets the pattern. The Firebase Web SDK already loads JS chunks via dynamic import; adding `gtag.js` as `next/script` with `strategy="afterInteractive"` is a separate network call and adds ~30 KB compressed.
- **`next/script` precedents.** Zero usages of `next/script` in source. The reCAPTCHA script is wrapped by `react-google-recaptcha`'s component (`src/components/recaptcha/ReCaptchaWrapper.tsx:30`), which loads its own script tag — that's not directly applicable to `gtag.js`. Status: **absent**. Severity: **nice-to-have**.
- **Heavy client init in layout.** `app/[locale]/layout.tsx:38-59` mounts (in order): `NavigationProgressBar`, `AuthInit`, `QuickRecommendButton`, `BaseSiteInit`, `AppInit` (which itself mounts 8 child initializers including `ChatsInit`, `PushInitializer`, `ChatsWatcher`, `InitFavoritesStore`, `UseTokenRefresh`, `AccountStateDialogsInit`, `SyncCardSizeFromCookie`, `ScrollToTop`), `DrawerDialogManager`. Status: **present**. Severity: **friction**. Why: a GA4 init component (consent read + `gtag` config + page-view firing) is one more sibling in this tree. Order matters: consent state must be read before the GA4 init publishes a `gtag('consent', 'default', …)` call. Slot it after `BaseSiteInit` and before the consent banner mount.

### 10. Cross-repo seams

Documented for Mastermind. No backend changes are expected this audit — the seams below describe what the feature spec must capture as assumed.

- **`sign_up` vs. `login` differentiation for social providers.** Backend currently does not return an `isNewUser` flag on the `firebase-sync` response (audit did not read backend; inference from `authService.ts:syncUserToBackend()` consumer at `useAuthStore.ts:294-296`). If Mastermind wants `sign_up` to fire for Google/Facebook first-time sign-ups (today they land in `loginWithGoogleFirebase` and `loginWithFacebookFirebase` identically to returning logins), the backend response shape must add an `isNewUser` boolean for the frontend to read. Web-only workaround: rely on Firebase's `additionalUserInfo.isNewUser` from `getAdditionalUserInfo(userCredential)` at the sign-in result — that's a frontend-only solution and does not require a backend change. Confirm which approach the feature spec wants.
- **`product_create_completed` success signal.** Web fires this on `result.type === 'success'` from `createNewProduct()` (`UploadedProductDialog.tsx:62`). The response shape returned by `POST /secure/products/create` is consumed as-is; the audit assumes the response carries at least the new `productId` so it can be attached to the event. No backend change needed if the response already returns the id.
- **`message_sent` success signal.** Web fires this on a non-null return from `useChatStore.sendMessage()` (`useChatStore.ts:580`), which is a Firestore write — not a backend HTTP roundtrip. Backend is uninvolved. The seam is `oglasino-web` ↔ Firestore, not `oglasino-web` ↔ backend. The instrumentation must fire only on Firestore commit success per the known silent-failure path.
- **`contact_seller_clicked`.** No backend roundtrip; pure web-side click. The only cross-repo seam is the `UserInfoDTO` shape supplied by the backend on product detail responses — `firebaseUid` is the safe identifier and is already present.
- **GA4 measurement ID environment binding.** Web reads `NEXT_PUBLIC_GA4_MEASUREMENT_ID` (proposed) from Vercel env. No backend or router changes. The Cloudflare Worker (`oglasino-router`) does not need to know about GA4 — its only relevant concern would be if Mastermind wanted to block `googletagmanager.com` in maintenance mode, which is out of scope for v1.

Three plausible answers: (a) one page-view per `slugs` path (a Saturday-morning user adjusting filters counts once), (b) debounced per searchParams change (more conventional for ecommerce filtering), (c) every change (most signal, noisiest in GA4). Pick one before instrumentation.
  2. **`sign_up` vs. `login` for social providers.** Today both land in the same Firebase-credential code path. Two options: read `getAdditionalUserInfo(userCredential).isNewUser` at the credential-result site (web-only, zero backend change), or have the backend return `isNewUser` on `firebase-sync` (cleaner, but adds a backend ask). Confirm.
  3. **`form_submit_failed` scope.** Product forms use the `{field, code, translationKey}` server contract; auth forms use Firebase error codes; message and review forms have no contract surface. Either (a) scope `form_submit_failed` to "product-form server-validation failures only" (clean, narrow), or (b) normalize auth/message surfaces to the same contract before instrumentation (clean, expensive).
  4. **`select_promotion` scope.** There is no card-level promoted-vs-organic distinction today. Either (a) narrow v1 `select_promotion` to free-zone-landing-page CTAs only (small but real), (b) wait on a future promoted-card surface, or (c) drop the event from v1 and pick it up in v2.
  5. **Consent banner mount coverage.** `CookieBanner` is mounted at `(portal)` and `owner/` layouts but not at `admin/` — confirm whether that is intentional. For Consent Mode v2, every public landing must have the banner; admin is auth-gated and lower-risk.
  6. **Authenticated-user consent persistence.** Today only `allowPreferenceCookies` is mirrored to the backend user record. If analytics consent should persist across devices for signed-in users, the backend `UserDTO` and `updateUserAuth` payload need an `allowAnalyticsCookies` field. Decide whether to include this in v1 or accept device-local persistence only.
  7. **GA4 transport.** Firebase Analytics via `firebase/analytics`'s `logEvent` (uses the same `gtag` under the hood, no separate script needed since Firebase loads it) vs. `next/script` loading `gtag.js` directly (explicit, decoupled from Firebase). The brief says "client-side via `gtag.js`" — interpreting that literally means option 2, but option 1 is functionally equivalent and reuses the Firebase init. Confirm interpretation.
