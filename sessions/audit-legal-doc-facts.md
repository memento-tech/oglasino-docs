## Q9(a) — Analytics gating + Advertising ID

### (a)(1) Auto-collection OFF by default until consent + runtime enable tied to consent

**Verdict: PARTIAL.** The runtime enable call tied to consent is **VERIFIED-TRUE**. A startup **config flag** disabling auto-collection is **NOT FOUND** — the "off by default at startup" guarantee rests entirely on JS, and the JS disable path does not run on a fresh/denied install.

**Runtime enable call tied to the consent setting — VERIFIED-TRUE:**

- `src/lib/consent/analyticsGate.ts:17-23` — `isAnalyticsConsentGranted()` returns `true` only when the persisted decision has `analytics === true`; absent/unhydrated/null/throw → `false` (default-deny).
- `src/lib/analytics/analyticsClient.ts:38-69` — `syncAnalyticsState()` reads the gate; **gate open** → lazy `getAnalytics()` then `setAnalyticsCollectionEnabled(instance, true)` (line 65); **gate closed** → only calls `setAnalyticsCollectionEnabled(instance, false)` **if already `initialized`** (lines 41-50).
- `src/components/init/AnalyticsInit.tsx:21-33` — runs `syncAnalyticsState()` once at launch and re-runs it whenever the consent decision flips (subscription), so toggling off stops collection without a restart.

**Config flag disabling auto-collection at startup — NOT FOUND:**

- No `firebase.json` at repo root (`ls firebase.json` → absent), so no `analytics_auto_collection_enabled: false`.
- `android/app/src/main/AndroidManifest.xml` — no `firebase_analytics_collection_enabled` meta-data.
- `ios/OglasinoDev/Info.plist` — no `FIREBASE_ANALYTICS_COLLECTION_ENABLED` key.
- Repo-wide `rg` for `analytics_auto_collection_enabled|firebase_analytics_collection_enabled|FIREBASE_ANALYTICS_COLLECTION_ENABLED|analytics_collection_deactivated` (excluding `node_modules`, `Pods`) → **zero hits.**

**Why this is PARTIAL, not TRUE:** `@react-native-firebase` native Analytics auto-collection defaults to **enabled**; it is suppressed only by one of the config flags above (none present) or by a persisted `setAnalyticsCollectionEnabled(false)`. On a **fresh install with no consent decision**, the gate is closed, so `syncAnalyticsState()` hits the `!gateOpen` branch but `initialized` is still `false` — it returns **without** calling `setAnalyticsCollectionEnabled(false)` (`analyticsClient.ts:41-50`). Same on an **explicit deny**. So no JS call ever writes the "off" override on those paths, and the native SDK runs at its default (enabled) until the user grants consent. The PP claim "OFF by default until in-app consent" is enforced for the *JS event-logging seam* (`track.ts` no-ops until `getAnalytics()` runs) but is **not** backed by a native-layer kill switch.

**One-sentence answer:** Consent-gated runtime enabling is correctly wired (`analyticsClient.ts:65` behind `analyticsGate.ts:17`), but there is no Firebase auto-collection-disable config flag anywhere, and the JS does not write the "off" override on a fresh-or-denied install — so native auto-collection is not provably OFF-by-default at startup.

### (a)(3) iOS — no AdSupport / ATT linkage in config or pods

**Verdict: VERIFIED-FALSE for the pods half (AdSupport IS linked); VERIFIED-TRUE for ATT (no ATT linkage).**

**AdSupport linkage IS present (contradicts the claim):**

- `ios/Podfile.lock:2846-2851` — `RNFBAnalytics (24.0.0)` depends on **`FirebaseAnalytics/IdentitySupport (= 12.10.0)`** (line 2851), the IDFA-enabled analytics variant.
- `ios/Podfile.lock:387-390` — `GoogleAppMeasurement/IdentitySupport` is the resolved measurement pod (the `IdentitySupport` subspec is the build that links `AdSupport.framework` / collects IDFA).
- `ios/Pods/GoogleAppMeasurement/Frameworks/GoogleAppMeasurementIdentitySupport.xcframework` is present on disk.
- The opt-out (`$RNFirebaseAnalyticsWithoutAdIdSupport = true`, which switches RNFBAnalytics to `FirebaseAnalytics/Core` without IDFA) is **NOT set** — `rg -i "RNFirebaseAnalyticsWithoutAdIdSupport|WithoutAdId|AdIdSupport" ios/Podfile` → zero hits.

**ATT (App Tracking Transparency) — VERIFIED-TRUE (no linkage):**

- `rg -i "NSUserTracking|AppTrackingTransparency|ATTrackingManager|requestTrackingAuthorization"` across `ios/` (excl. Pods), `src/`, `app/`, `app.config.ts` → **zero hits.** No `NSUserTrackingUsageDescription` in `Info.plist`. The code comments themselves state "no IDFA/ads, so no App Tracking Transparency gate" (`analyticsClient.ts:16-18`, `AnalyticsInit.tsx:13-14`) — but the linked pod variant contradicts the no-IDFA premise.

**One-sentence answer:** No ATT API or usage string is linked, but the iOS build **does** link the IDFA-capable `FirebaseAnalytics/IdentitySupport` + `GoogleAppMeasurementIdentitySupport` pods (the AdId opt-out flag is not set), so the "no AdSupport linkage in pods" claim is false as the project currently resolves.

## Q9(c) — In-app account deletion reachable

**Verdict: VERIFIED-TRUE.**

- Entry point: `app/owner/user.tsx:368-369` — a red "delete account" button (`onPress={() => openDialog(DialogId.DELETE_ACCOUNT_CONFIRMATION_DIALOG)}`) inside the owner profile/settings screen, under a "delete.account" section (lines 357-364), reachable from the user menu.
- Dialog wiring: `DialogId.DELETE_ACCOUNT_CONFIRMATION_DIALOG = 'deleteAccountConfirmationDialog'` (`src/components/dialog/dialogRegistry.ts:23`), registered to the component at `src/components/dialog/DialogManager.tsx:15,56`.
- Flow: `src/components/dialog/dialogs/DeleteAccountConfirmationDialog.tsx` — re-authenticates (password or Google, lines 71-97), then `deleteCurrentUser()` (line 133), sets `accountJustDeleted` (line 137), `auth.signOut()` + `router.push('/')` (lines 139-141).
- Service: `src/lib/services/userService.ts:66-74` — `deleteCurrentUser()` → `BACKEND_API.post('/secure/user/me/delete')`, returns `scheduledDeletionAt` (a **scheduled/soft delete** with a restore window — consistent with the `PENDING_DELETION` peer state and the `delete.account.restore.note` copy at `user.tsx:364`).

**One-sentence answer:** Yes — a reauth-gated account-deletion flow is reachable from the in-app owner settings screen (`app/owner/user.tsx:368`) and calls `POST /secure/user/me/delete` (`userService.ts:68`), satisfying PP §8.3 and Apple 5.1.1(v).

## Q9(d) — Push-token lifecycle on sign-out

**Verdict: VERIFIED-TRUE (client detaches server-side on sign-out); backend pruning out of repo.**

- On sign-out, `user` becomes `null`; `PushNotificationsInit.tsx:157-168` reacts to the user change and, when `currentUserId` is null, calls `unregisterFromPush()` (line 166).
- `src/notifications/lib/pushNotificationRegister.ts:45-58` — `unregisterFromPush()` calls `detachPushTokenFromBackend(currentToken)` (line 48) and clears the module token + listeners. (Guard: early-returns if `currentToken` is null, line 46 — i.e. it only fires when a token was registered this session.)
- `src/notifications/service/pushTokenService.ts:10-20` — `detachPushTokenFromBackend()` → `BACKEND_API.post('/secure/push/token/detach', { pushToken: token })` (errors intentionally swallowed).
- A second detach path exists for permission revocation while running: `pushNotificationRegister.ts:72-85` (AppState→active re-check).

**Out of repo (stated explicitly):** Backend-side pruning of stale tokens via Expo `DeviceNotRegistered` receipts lives in `oglasino-backend`, not this repo — not audited here.

**One-sentence answer:** On sign-out the app detaches the push token server-side via `POST /secure/push/token/detach` (`pushTokenService.ts:14`, triggered from `PushNotificationsInit.tsx:166`); any `DeviceNotRegistered`-receipt pruning is a backend concern outside this repo.

The same cleanup runs on the Firebase `onIdTokenChanged(null)` branch (`authStore.ts:258-265`).

**Intentionally survives sign-out** (persisted AsyncStorage keys never touched by logout — verified by reading each module and confirming none is referenced in `clearUserScopedStores`):

## Adjacent findings

1. **Analytics native default-on gap (severity: high — privacy/compliance).** `analyticsClient.ts:41-50` never writes `setAnalyticsCollectionEnabled(false)` on a fresh-or-denied install (the SDK is never `initialized` on those paths), and no `firebase.json`/manifest/plist auto-collection flag exists. The PP "analytics OFF by default until consent" claim is enforced only at the JS event-logging seam, not at the native Firebase Analytics layer. Out of audit scope to fix — flagging for the compliance review and a possible follow-up brief (add `firebase.json` `analytics_auto_collection_enabled:false` + `app_measurement` AdId-off, or set the disable in the gate-closed path unconditionally).

2. **iOS IDFA pod variant vs. PP "no advertising identifier" claim (severity: high — privacy/compliance).** `RNFBAnalytics` resolves `FirebaseAnalytics/IdentitySupport` (Podfile.lock:2851) because `$RNFirebaseAnalyticsWithoutAdIdSupport` is unset. This links the IDFA/AdSupport-capable measurement framework even though the app intends first-party-only analytics. Recommend setting the opt-out in the Podfile (and re-prebuilding) so the iOS binary links `FirebaseAnalytics/Core` instead. Out of scope to change (`ios/Podfile` is generated native config; also a hard-rule config-touch boundary).

3. **`google_analytics_adid_collection_enabled` never set on Android (severity: medium).** Even after consent, nothing disables Advertising-ID collection on Android (no config flag anywhere). If the PP promises no Ad-ID *ever*, this needs a manifest meta-data or `firebase.json` setting. Pairs with finding 1.

4. **Prebuild on disk is stale vs. `app.config.ts` (severity: medium — verification hazard).** `android/`+`ios/` are from 2026-06-05; the image-picker permission-string fix and any other 2026-06-09 `app.config.ts` changes are not reflected. Any on-device permission/string verification must run against a fresh prebuild or it will validate stale output. (Already tracked in `state.md` as the permissions on-device verification owed item.)

5. **`detachPushTokenFromBackend` swallows all errors (severity: low).** `pushTokenService.ts:16-18` silently ignores detach failures. Acceptable for best-effort cleanup, but a transient network failure on sign-out leaves the token attached until backend `DeviceNotRegistered` pruning catches it — worth noting for the PP push-token claim's completeness.
